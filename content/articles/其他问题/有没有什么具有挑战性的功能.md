##### ==大屏数据加载==

1. 在做社区全域大屏时，我们实现了完全的配置化。大屏由‘容器组件 -> 指标组件 -> 业务业务组件’三层嵌套组成。难点在于，所有业务组件都是异步加载的，且必须由外层的‘指标组件’统一调度数据请求。如果业务组件还没渲染完成，外层就调用其内部方法，会产生报错；但如果在每个业务组件里各写各的初始化逻辑，又会导致代码冗余，难以维护。
1. 为了实现**无侵入式的组件调度**，我设计了一套生命周期钩子监听方案。在 Vue 2 中利用 `@hook:mounted`，Vue 3 中利用 `@vue:mounted`进行组件生命周期监听。我在‘指标组件’这一层动态监听子组件的挂载状态。 这样，业务组件只需要保持‘纯净’的 UI 展示，所有的**加载时机、数据获取**都由外层的调度器统一控制。这不仅解决了报错问题，还能更好的进行组件复用，真正实现了配置化下的‘即插即用’

##### 大屏布局

1. 大屏开发中常遇到布局冲突。比如某个业务组件自带一个或‘筛选’按钮，按照 UI 设计，这个按钮必须显示在最外层大屏模块的右上角。由于可能会受到缩放的影响，传统的position定位模式可能导致定位偏差
1. 我没有采用传统的 CSS 强行定位，而是引入了 **Teleport（传送门）** 思想。 我在顶层模块定义了**‘插槽占位点’**，并利用 `Provide / Inject` 将这些占位点的 DOM 引用向下传递。深层的子组件通过 `Teleport` 将其交互控件精准地‘传送’到预定的 DOM 位置。 这种做法的妙处在于：**逻辑归子组件管（数据流清晰），渲染归父组件管（布局规整）**。这解决了大屏适配缩放下的坐标偏移问题，也保证了 HTML 结构的语义化，极大地降低了 UI 还原的复杂度。

##### 单兵系统接入

1. 在全域大屏项目中，我们需要集成三方单兵通讯系统，实现实时视频和对讲功能 。但对方只提供了一套基于 **jQuery** 开发的、逻辑极其耦合的 Demo 示例。我们需要在 Vue 3 架构中平滑接入，且面临三大难题：一是逻辑耦合，且代码未抽离，导致回调地狱。二是必须手动管理复杂的登录、保活流程；三是通讯链路极长，任何一个环节失败都会导致通话崩溃
2. 我通过Promise把需要的功能单独拆出，进行注释，使代码清晰明了，不必再进行嵌套
3. 首先把登录逻辑拆出，让其只在需要的时候进行登录，且只登录一次，初始化一次保活内容，避免性能浪费和内存的溢出
4. 然后对对讲逻辑和视频逻辑单独进行组件封装，使得在使用的时候只需要引入组件，传入设备信息就能进行通信和控制
5. 由于通讯链路过长，且每一步都都需要依托于上一步的成功，我还加入了失败重试的策略，哪一步失败就重试一定次数，确保通讯链路打通
6. 最终，我成功地将这套陈旧的系统整合进了全域驾驶舱，实现了大屏与单兵人员的对讲和实时视频查看动能。而且由于进行了封装，其他项目在用的时候可以直接复用

##### 打包体积优化

1. 可视化分析（定位问题）
   - 首先通过工具明确体积的大头在哪里。
2. 代码层面的“减量”（从源头控制）
   - **Tree Shaking：** 确保生产环境开启
   - **按需引入：** 对于像 `lodash`、`ant-design` 等库，使用 `lodash-es` 替代 `lodash`，并配置插件实现按需加载，避免全量引入。
   - **公共依赖清理：** 定期检查 `package.json`，清理废弃的依赖包，以及避免将开发工具类（devDependencies）误打包进生产包。
3. 构建产物的“拆分”（利用缓存与并发）
   - **代码分割（Code Splitting）：** 将 `node_modules` 下的第三方库单独打包为一个 `vendor` 包，因为它们不常变动，利用浏览器缓存。
   - 业务代码配合路由懒加载（`import()`），将大型组件拆分为独立 Chunk。
   - **动态导入（Dynamic Import）：** 对非首屏逻辑使用懒加载，仅在触发时才加载对应 JS。
4. 交付层面的“加载策略”（提升感知性能）
   - **压缩与混淆：** 打包时通过配置移除所有 console 和注释。
   - **静态资源预处理：** 小图片使用 `url-loader` 转为 Base64，大图片通过压缩工具优化。
   - **Gzip：** 在服务端配置 Nginx，对文本资源进行 Gzip 处理。
