1. **概念定义**：虚拟 DOM 是在内存中维持的一层轻量级 **VNode 节点树（代表数据状态的普通 JS 对象）**。它是真实 DOM 的一层抽象映射沙箱。

2. **从绝对性能的纯粹微观视角来看，虚拟 DOM 永远不可能比一个精细化、人肉调优到极致的原生 DOM 操作更快。** > 因为虚拟 DOM 并没有消灭真实 DOM 操作，它最终依然要调用 `document.createElement` 等原生方法。相比于原生操作，它在中间链路还凭空多出了一层**‘虚拟树构建、树对象递归比对、Map/哈希表映射查找’**的额外 JS 算力耗损。”

3. 为什么还要用它

   1. **平摊恶劣代码的反向开销（提供性能下限保障）**：在复杂业务中，普通开发者极易写出极其低效的代码（如在循环中频繁读取几何属性引发同步布局回流，或者高频无脑重绘整棵 DOM 树）。而虚拟 DOM 通过其内部的“异步批处理（Async Batching）”与 **“Diff 算法的脏检查属性合并”**，把多次杂乱的写操作在内存中合并，一次性、增量地 Patch 到浏览器，将页面的渲染开销从野蛮的“取决于乱写的代码”成功收拢到 **“可控的高内聚 $O(n)$ 范围内”**。

   2. **将心智模型从「过程驱动」彻底解放至「声明式驱动」**：它让开发者不需要再肉眼去追踪‘哪个状态变了需要修改哪个物理节点的哪个 class’，将开发心智完全统一为 $UI = f(state)$。这极大降低了大型工程的维护和重构成本。

   3. **跨端（Cross-platform）能力的底座支撑**：

      虚拟 DOM 将前端组件的逻辑层与宿主平台的底层像素绘制进行了**彻底解耦**。由于 VNode 只是纯粹的普通 JS 对象，通过更换底层的 `Renderer`（渲染器），该虚拟树既可以在浏览器中映射为 HTML DOM，也可以在移动端（Weex、Uni-app）被解析为 iOS/Android 的原生 Native 微件，甚至在 Node.js 服务端完成 `SSR`（服务端渲染）字符串拼装，成为跨端生态的物理基石。

4. Vue 3 认为“纯运行时”的虚拟 DOM Diff 太过昂贵（因为哪怕页面里只有一个文本在变，传统算法也会递归扫完几万个静态节点）。

   - **静态提升 (Static Promotion)**：编译期发现节点上的属性是写死的，会直接将其抽离提升至 Render 函数外部常驻内存。重新 Patch 时，这些节点在内存中仅保留一个引用，**Diff 算法无脑跳过**。

   - **区块树 (Block Tree) 与 动态标记 (Patch Flags)**：

     在动态节点的 VNode 上打上哈希数字标记（如 `1` 代表 Text 变动，`2` 代表 Class 变动）。在更新时，算法不需要深度递归树，而是通过一条**扁平的动态节点追踪链表（Dynamic Children Array）**，直接做到 $O(1)$ 复杂度的**靶向 Patch 属性硬修补**。

   - **结论**：Vue 3 将虚拟 DOM 的运行时开销，强行优化到了“完美正比于动态变量的个数，而与 HTML 模板的大小完全去耦”的境界。
