1. 当一个**对象（引用类型）\**需要转换为\**基本类型**时，ECMAScript 引擎会自动触发 `ToPrimitive` 抽象操作。

   - **参数行为**：`PreferredType` 是期望转换的目标类型（可选值为 `Number` 或 `String`）。
   - **运作流水线（洋葱逐层求值）**：
     - **Case A：如果指定为 `Number`**（例如参与数学运算、比对）：
       1. 优先调用对象的 **`valueOf()`** 方法，如果返回基本类型值，则采用。
       2. 否则，调用对象的 **`toString()`** 方法，如果返回基本类型值，则采用。
       3. 如果两步结束依然没有返回基本类型，直接抛出 `TypeError` 恶性线上故障。
     - **Case B：如果指定为 `String`**（例如模板字符串拼接、对象作为对象的 key）：
       1. 优先调用对象的 **`toString()`** 方法，成功则返回。
       2. 否则，反向调用对象的 **`valueOf()`** 方法，成功则返回。
       3. 若全部沦陷，同样抛出 `TypeError`。
     - **Case C：默认缺省（Default）**：除了 `Date` 对象默认倾向于 `String` 外，其余所有内置对象（包括普通的 `{}` 和数组 `[]`）默认全部倾向于按 `Number` 流程进行。

2. 其余三个高内聚基本类型转换规范

   - **`ToBoolean`**：除了以下 **7 个特殊的 Falsy（假值）** 固定转化为 `false` 外，其余全量标量和重型对象（包括空数组 `[]` 和空对象 `{}`）在内存中一律强行映射为 `true`：
     - `undefined`、`null`、`false`、`0`、`-0`、`NaN`、`""`（空字符串）。
   - **`ToNumber`**：
     - `undefined` > `NaN`
     - `null` > `0`
     - `""`（空字符串） > `0`
     - `"123"` > `123`，但 `"12a"` > `NaN`
   - **`ToString`**：普通基础标量直接加双引号。

3. 运算

   1. 加法特殊规则：如果 `+` 运算符的左右两边**任意一边存在字符串**，或者经过 `ToPrimitive` 转化后有字符串，则该行为退化为**纯粹的字符串拼接**

      ```js
      1 + "2"          // "12"
      1 + null         // 1 (null 触发 ToNumber 变为 0)
      1 + undefined    // NaN (undefined 触发 ToNumber 变为 NaN)
      [1, 2] + [3, 4]  // "1,23,4"
      ```

   2. 非加法算术运算符（`-`, `*`, `/`, `%`）：左右两边无脑一律执行 `ToNumber` 底层转换

      ```
      "5" - 2          // 3
      [5] - 2          // 3 ([5] -> "5" -> 5)
      ```

   3. 双等号 `==` 宽松相等判定根据 ECMAScript 官方规范，双等号 `==` 比对的运行时处理有着严格的**降维规约链**：

      - **规则 A：`null` 与 `undefined` 特权隔离**

        `null == undefined` 恒成立，且它们在 `==` 判定中**拒绝与除自身以外的任何其他数据类型发生相等转换**（比如 `null == 0` 是 `false`）。

      - **规则 B：基本类型 vs 基本类型 $\rightarrow$ 统一降维至 `Number`**

        数字比对字符串，或者布尔值比对任意类型，布尔值或字符串先通过 `ToNumber` 转为数字。

      - **规则 C：对象 vs 基本类型 $\rightarrow$ 唤醒 `ToPrimitive`**

        对象一侧率先执行 `ToPrimitive` 转换为基本类型，随后继续应用上述基本类型比对军规。
