1. `key` 是 VNode（虚拟节点）的唯一标识（UUID）。它直接服务于 **`patch` 阶段的 Diff 算法**
2. 利用 `key` 的唯一性去执行**声明式刷新**
3. 当虚拟节点【匹配到】时（Hit）
   1. **物理 DOM 留任**：旧的真实 DOM 节点完好无损地保留在浏览器中，**绝对不会**被销毁重建。
   2. **精细化属性 Patch**：框架只对比新旧 VNode 的差异（如：类名 `class` 变了、文字内容变了、绑定的点击事件变了），然后精准地调用原生的 `setAttribute` 或修改 `textContent`。
   3. **DOM 位移指针移动**：在长列表（如 Vue 3 的快速 Diff 算法或 React 的递增遍历）中，如果节点只是物理位置发生了改变，算法会计算出**最小移动距离**，通过原生的 `insertBefore` 执行物理位移，而不会动用昂贵的创建动作。
4. 当虚拟节点【匹配不到】时（Miss）
   1. **物理 DOM 彻底毁灭**：旧的真实 DOM 节点连同它底下的整棵子树，会被彻底从浏览器的 DOM 树中执行 `removeChild` 移除。
   2. **垃圾回收（GC）开销**：由于大面积的 DOM 节点和绑定的 JS 闭包被解绑，会被直接抛给浏览器的垃圾回收器（GC），引发短时间内的堆内存数据整理压力。
   3. **重新经历像素流水线**：框架会调用原生的 `document.createElement` 重新创建一整棵全新的真实 DOM 树并将其插入页面，浏览器必须重新走一遍 **解析 > 布局/回流 > 重绘 > 合成** 的全量像素流水线。
   4. **生命周期大换血**：原有的组件实例被彻底注销，触发 `beforeUnmount` 与 `unmounted` 钩子。随后新组件被激活，**完整地重新经历一遍** `beforeCreate` > `created` > `mounted` 初始化洗礼，**所有历史内存变量和状态彻底清空、归零**。
5. 如果使用id当key，就会保证节点的正确匹配，不会产生这么大的开销
6. 如果使用index，在一些只展示的数据或者只在末尾进行插入的列表不会出现问题，但是对于一些需要排序和插入的内容，会导致每个节点的key错误匹配，原本可能只需改变一点内容或者不用改变的内容，需要整个内容进行改变
7. 只进行同层节点的比对。`key` 的唯一性只需要在当前同级兄弟节点（Siblings）中得到保证。不同物理层级、或者不同父节点下的子节点，即使配置了完全相同的 `key` 值，也绝对不会发生任何冲突，更不会被框架误复用。
