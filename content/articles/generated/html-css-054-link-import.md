| **维度**           | **<link>**                                                   | **@import**                                                  |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **所属范畴**       | HTML 规范（原生标签）                                        | CSS 规范（内部指令）                                         |
| **网络加载行为**   | **并行下载**，与 DOM 解析互不阻碍，提速白屏指標。            | **串行下载**，必须等宿主文件解析完才触发，加剧网络瀑布流死锁。 |
| **DOM 交互与操作** | **支持**。可以通过 JavaScript 动态操作 DOM 改变 `href`，运行时无痛切率样式。 | **不支持**。由于纯缩于 CSS 内部，JS 无法直接定位或动态修改 `@import` 的路径。 |
| **兼容性边界**     | 零兼容包袱（自 HTML 诞生的底座标签）。                       | 古董浏览器（如 IE5 以下）无法解算，现代已无此包袱。          |
| **特异性权重**     | 遵守标准的后来者居上与特异性层叠。                           | **强制提升物理最前线**。W3C 规范要求 `@import` 必须写在 CSS 文件的**最顶部**，否则直接失效。 |

1. 既然 `@import` 有这么严重的网络阻塞缺点，为什么我们在现代 Vue/React 项目（如 Sass/Less/PostCSS）里还天天写 `@import` 或者是 `@use` 呢？
   1. “这直接涉及**‘源码期 `@import`’与‘运行时 `@import`’的物理隔离**。
   2. 在我们在开发中写的 `@import "variables.scss"`，它属于 **CSS 预处理器的编译语法，而非浏览器的原生指令**
   3. 编译引擎会在内存中通过**AST（抽象语法树）解析，直接将这些被引入的代码物理合并、平铺到当前文件中，最终编译产出的是一张完全没有原生 `@import` 痕迹的、干净的单一 CSS 产物**
