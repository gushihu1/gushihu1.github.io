1. **堪称完美的“硬隔离”沙箱（零污染）**
   - **样式隔离**：`iframe` 拥有独立的 DOM 树和 CSSOM 树。主应用与子应用之间的样式不论怎么写（即使都是全局 `body { background: red }`），也**绝对不会发生样式冲突**。
   - **JS 脚本隔离**：每一个 `iframe` 拥有自己独立的 `window` 全局执行上下文。子应用污染、修改全局变量（如覆写 `window.Array.prototype` 或挂载全局全局变量）完全被限制在自己的“沙盒”内，对主应用毫无影响。
   - **加分表达**：这是目前微前端框架（如 Single-SPA 体系、qiankun）通过 JS 代理沙箱（Proxy Sandbox）或快照沙箱苦苦追求的完美隔离状态，而 `iframe` 在浏览器 C++ 内核层天生就完美支持。
2. **跨技术栈/老旧项目集成的“终极解耦”**
   - 如果团队需要在一个现代的 React/Vue 3 项目中嵌入一个十年前用 jQuery、ExtJS 或 Angular 1 编写的巨石系统，或者集成完全由不同外包团队开发的、没有源码的资产应用，**`iframe` 是唯一能够不改动一行子应用代码、几秒钟实现低成本上线的方案。**
3. **天然的安全边界**
   - 在接入第三方非信任组件（如第三方支付、广告广告位、社交分享插件）时，配合 `iframe` 的 `sandbox` 属性（如 `sandbox="allow-scripts allow-forms"`），可以从底层限制子应用的执行权限（如禁止顶级弹窗、禁止运行脚本等），防止 XSS 攻击蔓延到主应用。
4. **UI 与交互体验的“割裂感”**
   - **弹窗居中与遮罩问题**：子应用的 Modal 弹窗或 Loading 遮罩只能居中在 `iframe` 的物理切片区域内，无法撑满整个浏览器屏幕。
   - **右键菜单与下拉框截断**：如果 `iframe` 尺寸固定或较小，内部组件（如 Select 下拉框、DatePicker 展开面板）超出边缘时会被物理截断，或者导致 `iframe` 出现极其难看的双滚动条。
5. **路由状态不同步（刷新白屏/丢失历史）**
   - `iframe` 内部的路由跳转不会自动同步更新到主应用的浏览器地址栏（URL）。
   - **致命体验**：一旦用户在子应用操作到了第 5 步，不小心按了浏览器的“刷新”键，整个主应用重新加载，`iframe` 内部直接回到了初始的第 1 步首页。同样，浏览器的“前进/后退”也会产生混乱。
6. **极高的性能开销与主线程阻塞**
   - **昂贵的加载成本**：创建一个 `iframe` 页面的开销相当于重新打开一个浏览器标签页。它会建立全新的 HTTP 请求，重新下载、解析、执行公共的依赖库（如 Vue、React、Element UI），导致严重的内存冗余。
   - **阻塞主应用 `onload` 事件**：默认情况下，`iframe` 的加载会和主应用的静态资源抢占浏览器并发连接数。如果子应用加载缓慢，会死死死死扣住主应用的 `window.onload` 触发，导致主应用的性能指标（如 LCP）直线下滑。
7. **通信与跨域成本高昂**
   - 如果主应用和 `iframe` 子应用不同源，只能通过 `window.postMessage` 进行异步的基于事件流的通信。这种机制无法做到像单页应用中共享全局状态机（如 Pinia/Redux）或直接进行同步的方法调用那样丝滑，需要编写大量的事件监听与派发管道代码。
8. **堆内存泄漏隐患**
   - 当主应用动态销毁包含 `iframe` 的 DOM 节点时，如果 `iframe` 内部还驻留有未清理的异步定时器、全局事件监听、或者主应用曾持有 `iframe.contentWindow` 的闭包引用，**该 `iframe` 的整个上下文及庞大的 DOM 树将常驻堆内存，无法被 V8 引擎回收**，极易引发页面越用越卡的内存泄漏。
